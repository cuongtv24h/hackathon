Artifact 1 — Solution Overview {#artifact-1}
1.1 Solution Summary
Trợ lý Thông tin Bệnh viện AI là hệ thống AI-native phục vụ bệnh nhân và người nhà Bệnh viện Tim Hà Nội. Hệ thống cung cấp khả năng tra cứu thông tin chính thức 24/7, hỗ trợ luồng đặt lịch hẹn và xử lý an toàn tình huống khẩn cấp thông qua giao diện hội thoại tự nhiên.

LLM là lớp tiếp nhận và xử lý trung tâm — phân tích intent, lập kế hoạch và gọi tools tương ứng. Keyword Pre-filter là safety net tốc độ cao cho critical emergency cases, không thay thế vai trò trung tâm của LLM.

1.2 Architecture Principles
#	Nguyên tắc	Diễn giải
AP-01	LLM-First	LLM là lớp tiếp nhận và xử lý trung tâm mọi request
AP-02	Safety Net	Keyword Pre-filter bảo vệ critical cases khi LLM không khả dụng
AP-03	Grounded by Design	RAG bắt buộc trước mọi generation — không hallucinate
AP-04	Separation of Concerns	Business logic, AI logic, data logic tách biệt rõ ràng
AP-05	Provider Agnostic	LLM Abstraction Layer với fallback chain — không phụ thuộc một provider
AP-06	Traceable	Mọi response có citation metadata truy xuất về knowledge base
AP-07	Graceful Degradation	Fallback chain: LLM Provider 1 → 2 → N → Hotline message
AP-08	Clean Extensibility	Code sạch, config-driven, dễ mở rộng multi-tenant sau

3.2 Keyword Pre-filter — Chi tiết Component
text

KEYWORD PRE-FILTER
├── Vị trí: Trong Chat Gateway (không phải module riêng)
├── Loại: Synchronous function, chạy trên mọi input
├── Thời gian: < 10ms (string matching local)
├── Không cần: LLM, network, database
│
├── INPUT: Raw user message text
│
├── PROCESSING:
│   Step 1: Normalize text (lowercase, bỏ dấu optional)
│   Step 2: Match against CRITICAL keyword list
│   Step 3: Match against CAUTION keyword list
│
├── OUTPUT 3 cases:
│   Case A — CRITICAL match:
│     {result: "critical", level: 2, matched: [...]}
│     → Chat Gateway BYPASS LLM
│     → Load Emergency Protocol Level 2 trực tiếp
│     → Response ngay (< 100ms tổng)
│
│   Case B — CAUTION match:
│     {result: "caution", keywords: [...]}
│     → Chat Gateway THÊM flag vào LLM context
│     → LLM nhận: {caution_keywords_detected: ["đau ngực"]}
│     → System prompt nhắc LLM: "Caution detected, evaluate carefully"
│     → LLM quyết định trigger_emergency() hay không
│
│   Case C — No match:
│     {result: "normal"}
│     → Chat Gateway chuyển cho LLM Orchestrator bình thường
│
├── CRITICAL KEYWORD LIST (ví dụ — cần bác sĩ phê duyệt):
│   NHÓM Tim mạch cấp:
│   - "đau ngực dữ dội", "đau ngực không chịu được"
│   - "tức ngực không thở được", "đau thắt ngực"
│   NHÓM Hô hấp cấp:
│   - "không thở được", "khó thở đột ngột", "nghẹt thở"
│   NHÓM Mất ý thức:
│   - "ngất xỉu", "bất tỉnh", "hôn mê", "mất ý thức"
│   NHÓM Đột quỵ cấp:
│   - "liệt nửa người", "méo miệng đột ngột"
│   - "không nói được đột ngột"
│
└── CAUTION KEYWORD LIST:
    - "đau ngực" (đơn lẻ, không có modifier)
    - "khó thở" (đơn lẻ)
    - "tim đập bất thường", "chóng mặt dữ dội"
    - "tê tay" + "nói lắp" (combination signals)

LLM ORCHESTRATOR — TRUNG TÂM CỦA HỆ THỐNG
│
├── INPUT nhận được:
│   ├── user_message: string

│   │   "Nếu phát hiện bất kỳ dấu hiệu nào cho thấy
│   │    người dùng hoặc người thân đang trong tình trạng

│   ├── GROUNDING RULE:
│   │   "Chỉ trả lời từ kết quả search_knowledge_base().
│   │    Nếu không có kết quả đủ tin cậy, gọi fallback_response()."
│   ├── MEDICAL REFUSAL RULE:
│   │   "Từ chối mọi yêu cầu chẩn đoán, tư vấn điều trị,
│   │    giải thích kết quả xét nghiệm."
│   ├── AVAILABLE TOOLS: [danh sách tất cả tools]
│   └── CITATION FORMAT: "Nguồn: [tên tài liệu], [trang]"
│
├── TOOL CALLING FLOW:
│   LLM phân tích → lập kế hoạch → gọi tool → nhận kết quả
│   → đánh giá → gọi tool tiếp (nếu cần) → tổng hợp response
│
└── OUTPUT:
    ├── streaming_tokens: text response
    ├── citations: [{source, page, domain}]
    ├── tools_called: string[]
    └── emergency_triggered: bool

Artifact 10 — Architecture Decision Record {#artifact-10}
ADR-001: Emergency Architecture — Approach C (Keyword Pre-filter + LLM Tool)
Field	Content
Context	Cần emergency detection đảm bảo an toàn, LLM là lớp chính
Problem	Emergency detection cần nhanh + reliable nhưng LLM có latency và có thể fail
Options	A: Emergency module riêng trước LLM (mâu thuẫn "LLM là lớp chính"); B: Chỉ LLM (phụ thuộc LLM); C: Keyword Pre-filter + LLM tool
Decision	Approach C
Reason	(1) LLM vẫn là lớp chính — nhất quán với thiết kế; (2) Keyword Pre-filter là safety net nhỏ (< 50 dòng code), không phải module lớn; (3) Critical keywords bypass LLM → < 100ms, không phụ thuộc internet; (4) LLM catch indirect descriptions mà keyword bỏ sót; (5) Caution flags bridge giữa hai layers
Trade-offs	Indirect descriptions (Level 1) phụ thuộc LLM — acceptable vì critical cases đã được keyword bắt
ADR-002: LLM là Lớp Tiếp nhận Trung tâm
Field	Content
Context	Cần quyết định ai là "brain" của hệ thống
Problem	Có thể có nhiều module xử lý trước LLM, làm phức tạp và duplicate
Decision	LLM là lớp tiếp nhận và xử lý trung tâm mọi request
Reason	(1) LLM có khả năng phân tích ngôn ngữ tự nhiên tốt nhất; (2) Tool calling là cơ chế mạnh để tích hợp; (3) Đơn giản hóa pipeline; (4) Keyword Pre-filter là exception nhỏ, không phải rule
Trade-offs	Phụ thuộc LLM availability — mitigate bằng fallback chain và Keyword Pre-filter safety net
ADR-003: LangGraph cho orchestration runtime
Field	Content
Decision	Dùng LangGraph cho bounded tool-calling graph, persistence và human-in-the-loop; không dùng intent router hoặc prebuilt autonomous multi-agent architecture
Reason	Graph có durable execution, checkpoint, streaming và interrupt/resume phù hợp booking confirmation; backend vẫn kiểm soát emergency, authorization, policy, grounding và audit
Trade-offs	Thêm dependency và yêu cầu quản trị checkpoint/version; graph state phải tối giản, bảo vệ PII và có retention
ADR-004: Supabase pgvector cho Vector Store
Field	Content
Decision	Supabase pgvector (PostgreSQL-native)
Reason	Managed service, một hệ thống duy nhất, đủ tốt cho ~200 chunks, SQL flexibility
Trade-offs	Không scale bằng dedicated vector DB khi KB lên triệu chunks
ADR-005: SSE cho Streaming
Field	Content
Decision	Server-Sent Events (không WebSocket)
Reason	Chat là unidirectional server→client; SSE đơn giản hơn, native browser support
Trade-offs	Không bidirectional — không cần cho chat
