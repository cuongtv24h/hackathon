# 6. AI Orchestration

## 1. Delivery Operating Model

| Role | Mission | Responsibilities | Boundary | Handoff |
|---|---|---|---|---|
| Human Lead | Giữ đúng scope và quyết định delivery | Approve exceptions, priority, merge/release decision, source-of-truth conflicts | Không coding thay builder trừ khi nhận ownership rõ ràng | Ready task → Builder; blocked/escalation → Lead |
| Human Dev | Cung cấp domain/implementation judgment | Confirm environment, review high-risk diffs, resolve product ambiguity | Không thay đổi contract/architecture một mình | Evidence hoặc decision → Lead/Reviewer |
| Builder Agent | Hoàn thành đúng một work package | Đọc pack, chỉ sửa allowed zone, chạy checks, nộp evidence | Không sửa ngoài task contract; không tự đổi spec | Reviewable branch → Reviewer |
| Reviewer Agent | Kiểm tra correctness và contract compliance | Diff/zone/marker/acceptance review; approve hoặc request fix | Không tự mở rộng scope; không merge release gate | Approved → Integration; findings → Fix |
| Fix Agent | Sửa finding có định danh | Chỉ sửa finding/retest trong cùng zone | Không tạo feature mới hoặc đóng finding thiếu evidence | Updated branch → Reviewer |
| Audit Agent | Đánh giá độc lập cuối sprint | Traceability, safety, RAG/PII/emergency, quality evidence | Không sửa builder branch trừ audit-fix được Lead giao | Audit result → Lead/Integration |
 
Handoff tối thiểu gồm branch/commit, work package ID, mandatory-read list, changed zones, commands đã chạy, kết quả và known limitations.

## 2. Repo Bootstrap Strategy

### 2.1 Chiến lược repository và branch

Dùng một monorepo vì component ownership đã chốt dưới `apps/`, shared contracts, Supabase, config và tests có dependency chặt. Default branch là `main`.

- Builder branch: `task/WP-xxx`.
- Fix branch: tiếp tục branch task hoặc `fix/WP-xxx-<finding-id>`.
- Integration branch tạm thời: `integration/sprint-<n>`.
- Không force-push sau khi review bắt đầu; rebase/cherry-pick do Integration owner thực hiện.
- Một branch chỉ có một work package; cross-task commit bị từ chối.

### 2.2 Scaffold và manifests

`docs/repo-manifest.yaml` là inventory workspace. `docs/file-ownership.yaml` định nghĩa exclusive-write zones. `docs/task-file-map.yaml` là high-level map; `docs/spec-registry/task-to-file-contract-map.yaml` là contract thực thi cho cả 40 WP.

`scripts/bootstrap.bat` chỉ tạo directory rỗng, idempotent và không overwrite. `scripts/verify-scaffold.bat` có hai gate:

1. Base verification sau WP-002/WP-003.
2. `/regions` verification sau WP-004, trước builder đầu tiên trong từng zone.

### 2.3 Scaffold version policy

Thay đổi manifest, ownership, region policy hoặc task-file map làm tăng patch version. Thay đổi zone/contract đang active cần Human Lead approval, cập nhật version và audit ảnh hưởng đến packets chưa chạy.

## 3. File Ownership & Task-to-File Contracts

- Mỗi WP có owner pool, branch, path thật, operation và marker expectation trong `docs/spec-registry/task-to-file-contract-map.yaml`.
- Specs chưa chọn ngôn ngữ/framework. Vì vậy `directory_zone` là contract path hợp lệ, không phải wildcard mơ hồ; Phase 6.5 ghi leaf file path trước coding.
- `FULL_FILE` chỉ cho governance file mới hoặc exception đã được Lead phê duyệt.
- Builder không sửa file/zone khác, kể cả khi thấy lỗi. Tạo finding/escalation thay vì sửa opportunistic.
- Region rules nằm tại `docs/region-marker-policy.md`.

## 4. PromptOps Repository Design

```text
prompts/
  _shared/
    README.md
  sprint-1/
    registry.yaml
    task-packets/     # Phase 6.5
    review/           # derived from packet + diff
    fix/              # derived from finding
    audit/            # sprint-level
```

Packet name: `WP-xxx.build.md`; review/fix/audit dùng cùng ID và suffix. Packet registry ghi pack version, branch, status, mandatory inputs, allowed zones và evidence links. Một task chỉ có một builder packet; reviewer/fix/audit không sửa packet gốc.

## 5. Board Workflow

GitHub Projects V2 dùng các cột bắt buộc:

| Column | Ý nghĩa | Entry | Exit |
|---|---|---|---|
| Prompt Backlog | WP đã dependency-ready nhưng chưa có packet | Upstream gate pass | Packet được Lead kiểm tra |
| Prompting & Gen | Builder đang thực hiện một packet | Packet + branch + zones có mặt | Builder gửi evidence, diff reviewable |
| Human Review (Audit) | Reviewer kiểm tra task; audit sprint có thể chạy song song | Checks tối thiểu pass | Approve hoặc finding có ID |
| Integration | Branch approved đang merge/chạy integration suite | Reviewer approve | Integration checks pass |
| Stable/Done | Đã merge và có evidence audit/gate phù hợp | Integration pass; audit gate nếu cuối sprint | Không tự quay lại trừ regression |

Allowed loops: Review → Prompting & Gen qua Fix; Integration → Prompting & Gen khi integration failure; Stable/Done → Prompt Backlog chỉ qua regression issue mới.

Labels: `wp:WP-xxx`, `pool:{data|api|ai-core|fe|qa|other}`, `wave:{0..7}`, `size:{s|m|l}`, `cap:{pc-01|pc-02|pc-03|pc-04|cross}`, `risk:{safety|pii|data|integration}`, `gate:{scaffold|review|audit|release}`.

## 6. Review / Fix / Audit Governance

### Definition of Ready

- Upstream dependencies done.
- Work package pack and task-to-file contract exist.
- Base scaffold verification passes; region verification passes where required.
- Packet has mandatory reads, acceptance criteria, allowed zone and verification commands.

### Definition of Reviewable

- One work package only; branch and diff available.
- All changed paths are contracted.
- Builder reports checks/evidence and known limitations.
- No secret, generated binary, unrelated reformat or unapproved marker change.

### Definition of Done

- Reviewer approved; required checks pass.
- Integration branch pass if task crosses components.
- Audit finding count is zero for sprint gate, or every accepted residual risk is signed by Human Lead.
- Registry/pack updates are included if implementation revealed an approved specification change.

### Review, fix and audit

Reviewer checks mandatory reads, contract/DTO compliance, zone/marker compliance, tests and acceptance criteria. Findings use `FIND-<WP>-<n>` with severity/blocker. Fix Agent only addresses identified findings and reruns affected checks.

Audit runs at the end of each proposed sprint: traceability sample, ownership drift, secret scan, RAG provenance/citation, emergency deterministic path, PII/logging, test evidence and deployment config. PC-02 and WP-606 always require Human Lead sign-off.

## 7. Merge & Integration Gates

Merge strictness is Medium:

- Mandatory: clean base/region verification, one reviewer approval, task acceptance checks, no ownership violation.
- Required before integration: upstream dependencies, contract checks for API/tool/DTO changes, relevant test evidence.
- Required before Stable/Done: integration suite for affected capability and audit result at sprint boundary.
- Hard blockers: secret committed, unapproved cross-zone edit, broken canonical contract, emergency bypassing pre-filter, ungrounded approved-source claim, PII leakage.
- Human Lead may accept a non-blocking residual risk only with recorded rationale and expiry.

### 7.1 48-hour sprint proposal

| Sprint | Duration | Scope / exit gate |
|---|---:|---|
| Sprint 0 | 4h | WP-001–004; base + region scaffold gates |
| Sprint 1 | 16h | WP-005–105; data, ingestion and foundation services |
| Sprint 2 | 16h | WP-201–506; tools, AI, APIs and experience lanes |
| Sprint 3 | 12h | WP-601–606; integration, safety, NFR and VPS release decision |

Five builders use parallel lanes only when DAG dependencies are satisfied. Reviewer/Audit capacity is reserved, not counted as a sixth builder lane.

## 8. Risks and Notes

- Technology stack is intentionally not invented: directory zones resolve safe ownership now; Phase 6.5 packets must declare leaf file path only after Lead confirms runtime/tooling.
- No task packet or GitHub issue body is generated in Phase 6.
- GitHub Projects V2 là target board, nhưng workspace hiện chưa có Git repository hợp lệ (thư mục `.git` không có metadata cần thiết). Human Lead phải clone hoặc khởi tạo repository/remote trước khi bật branch protection, board automation và merge workflow.
- Bootstrap data is `approved_for_pilot`; post-bootstrap content requires Content Management approval before it becomes answerable.
- Contact, emergency and production legal/retention decisions remain constrained by Phase 4/5; release gate cannot waive them silently.
